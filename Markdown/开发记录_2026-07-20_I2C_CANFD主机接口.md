# 2026年7月20日 I2C/CANFD 主机接口开发记录

## 开发背景

设备需要作为主机与从机通讯，本次先打通 I2C 和 CANFD 两条主机侧通讯接口。现有主机与上位机屏幕的通讯协议保持不变，RS422 / RS485 测试模式也保持原来的编译时选择方式。

本次开发目标不是做完整业务闭环，而是先把外设初始化、协议常量、主机收发 API、CANFD 响应轮询和 Keil 工程编译链路打通，为后续接寄存器表和屏幕命令做准备。

## 输入需求

从机协议要求如下：

- I2C 从机地址：`0x43`。
- I2C2 和 I2C4 共用同一套寄存器表。
- I2C 写：第 1 字节是寄存器地址，后续字节写入连续寄存器。
- I2C 读：从当前寄存器地址开始连续读。
- CANFD 命令 ID：`0x18EF4301`。
- CANFD 响应 ID：`0x18EF0143`。
- CANFD 数据格式：Byte0 为 `sequence`，Byte1 为 `command`，Byte2 及后续为参数。
- 支持命令：`0x01 PING`、`0x02 READ_REG`、`0x03 WRITE_REG`、`0x04 SET_CAN_MODE`、`0x05 CLEAR_FLAGS`。

硬件引脚要求：

- 打开 `PB8 / PB9` 配置 I2C。
- 打开 `PD1 / PD0` 配置 CANFD。

## 设计决策

### 1. I2C 使用 I2C1

主机侧把 `PB8 / PB9` 配置为 `I2C1`：

- `PB8` / `PB9`
- `GPIO_AF4_I2C1`
- 开漏复用输出
- 上拉

从机侧虽然描述为 I2C2 和 I2C4 共用寄存器表，但这是从机内部资源。主机侧只需要按地址 `0x43` 访问即可，不需要关心从机内部是哪一路 I2C 接口响应。

### 2. CANFD 使用 FDCAN1

主机侧把 `PD0 / PD1` 配置为 `FDCAN1`：

- `PD0 / PD1`
- `GPIO_AF9_FDCAN1`
- 推挽复用输出
- 高速 GPIO

FDCAN 只配置 1 个扩展 ID 过滤器，接收响应 ID `0x18EF0143`。其它帧全部拒收，避免现场总线上其它 CAN 帧污染本次调试统计。

### 3. 新增 `slave_comm` 应用层封装

新增 `slave_comm` 模块，专门承接主机到从机的 I2C/CANFD 通讯：

- 协议常量集中定义在 `slave_comm.h`。
- I2C 寄存器读写封装成 `SlaveComm_I2C_WriteRegs()` / `SlaveComm_I2C_ReadRegs()`。
- CANFD 命令封装成 `SlaveComm_CanPing()` 等 5 个接口。
- CANFD sequence 自动递增。
- CANFD 响应由 `SlaveComm_Poll()` 轮询接收。
- 统计变量和最后一帧数据保留为全局 `volatile`，方便 Keil Watch 观察。

### 4. 不修改屏幕协议

本次没有修改 `host_comm` 协议帧，也没有新增屏幕控件 ID。`HostComm_Poll()` 仍然只处理原来的上位机屏幕协议；`SlaveComm_Poll()` 只处理 CANFD 响应。

这样做的好处是：先确认从机通讯接口本身能编译、能初始化、API 清晰，后续再决定屏幕侧如何触发 I2C/CANFD 命令。

## 当前接口说明

### I2C 接口

```c
HAL_StatusTypeDef SlaveComm_I2C_WriteRegs(uint8_t reg,
                                          const uint8_t *data,
                                          uint16_t length);

HAL_StatusTypeDef SlaveComm_I2C_ReadRegs(uint8_t reg,
                                         uint8_t *data,
                                         uint16_t length);
```

当前限制：

- 最大连续传输长度：`64` 字节。
- 超时时间：`100ms`。
- HAL I2C 地址使用左移后的 `0x86`，协议文档中仍按 7 位地址 `0x43` 记录。

### CANFD 接口

```c
HAL_StatusTypeDef SlaveComm_CanPing(void);
HAL_StatusTypeDef SlaveComm_CanReadReg(uint8_t reg, uint8_t length);
HAL_StatusTypeDef SlaveComm_CanWriteReg(uint8_t reg,
                                        const uint8_t *data,
                                        uint8_t length);
HAL_StatusTypeDef SlaveComm_CanSetCanMode(uint8_t mode);
HAL_StatusTypeDef SlaveComm_CanClearFlags(uint16_t flags);
```

发送 ID 和响应 ID：

- 命令扩展 ID：`0x18EF4301`
- 响应扩展 ID：`0x18EF0143`

发送数据格式：

| 字节 | 内容 |
| ---- | ---- |
| Byte0 | sequence |
| Byte1 | command |
| Byte2.. | 参数 |

当前接收策略：

- `SlaveComm_Poll()` 轮询 FDCAN RX FIFO0。
- 只接收扩展 ID `0x18EF0143`。
- 其它帧计入 `slave_comm_can_ignored_count`。
- 最后一帧响应保存到 `slave_comm_last_can_data` 及相关 Watch 变量。

## 后续建议

后续现场联调建议按下面顺序推进：

1. 用示波器或逻辑分析仪确认 PB8/PB9 上是否有 I2C 起始信号、地址 `0x43` 是否 ACK。
2. 用 CANFD 分析仪确认 PD0/PD1 对应收发器侧是否发出扩展 ID `0x18EF4301`。
3. 和从机确认 CANFD 仲裁段/数据段实际波特率，如果不一致，优先修改 `Core/Src/fdcan.c` 中的位时序参数。
4. 拿到完整寄存器表后，再在 `slave_comm` 上层增加寄存器含义封装。
5. 从机基础通讯稳定后，再决定是否把屏幕控件命令映射到 I2C/CANFD 操作。

当前版本没有自动周期 PING，也没有自动读取寄存器；需要调用 `SlaveComm_CanPing()` 或 I2C/CANFD 读写接口才会主动发起从机通讯。
