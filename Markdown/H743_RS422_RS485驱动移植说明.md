# H743 RS422 / RS485 驱动移植说明

本文用于把本工程中的 RS422 底层驱动、RS422 协议层和 RS485 主机测试逻辑移植到另一套 STM32H743 工程中。目标工程建议同样使用 STM32CubeMX + STM32H7 HAL + Keil MDK 工程结构。

## 1. 当前代码结构

需要移植的核心文件如下：

| 模块           | 头文件                               | 源文件                          | 说明                                                          |
| -------------- | ------------------------------------ | ------------------------------- | ------------------------------------------------------------- |
| RS422 底层收发 | `Core/Inc/rs422.h`                 | `Core/Src/rs422.c`            | UART 中断收发、TX_EN 方向控制、环形缓冲、波特率切换和诊断计数 |
| RS422 协议层   | `Core/Inc/rs422_protocol.h`        | `Core/Src/rs422_protocol.c`   | 135 字节固定帧格式、包头同步、校验、协议统计                  |
| RS485 测速主机 | `Core/Inc/rs485_speed_test.h`      | `Core/Src/rs485_speed_test.c` | 基于 RS422 协议层的 RS485 主机轮询、从机地址统计              |
| RS485 载荷格式 | `Core/Inc/rs485_speed_test_wire.h` | 无                              | RS485 payload 内部字段偏移和命令字                            |

如果目标项目还要继续使用串口屏/上位机控制流程，再额外移植：

| 模块       | 头文件                   | 源文件                   | 说明                                                 |
| ---------- | ------------------------ | ------------------------ | ---------------------------------------------------- |
| 上位机通信 | `Core/Inc/host_comm.h` | `Core/Src/host_comm.c` | UART7 接收屏幕命令、启动/停止测试、上传 TX/RX/误码率 |

如果只移植 422/485 驱动，不移植 `host_comm`，需要在 `rs422.c` 里去掉 `#include "host_comm.h"`，并删除或改写 HAL 回调中的 `HostComm_UART_*Callback()` 转发调用。

## 2. 当前硬件映射

本工程当前映射如下，目标工程可以按自己的原理图改，但必须在代码中同步修改。

| 逻辑端口           | 当前外部口  | UART       | TX  | RX   | 方向脚             | 有效电平   |
| ------------------ | ----------- | ---------- | --- | ---- | ------------------ | ---------- |
| `RS422_PORT_U9`  | J12 / U9    | `USART1` | PA9 | PA10 | `TX_EN_1` = PA11 | 高电平发送 |
| `RS422_PORT_U10` | J13 / U10   | `USART3` | PD8 | PD9  | `TX_EN_2` = PD10 | 高电平发送 |
| 上位机通信         | 屏幕/上位机 | `UART7`  | PE8 | PE7  | 无                 | 无         |

注意：U10 当前确认接的是 `USART3`，不是 `USART2`。

RS485 主机模式当前只使用一条物理 485 总线：

- 物理总线固定为 `RS485_SPEED_TEST_PORT`，当前定义为 `RS422_PORT_U9`。
- 上位机“串口1”表示逻辑从机地址 `1`。
- 上位机“串口3”表示逻辑从机地址 `2`。
- `RS422_PORT_U10 / USART3` 在 RS485 双从机轮询版本中不是第二条独立物理 485 总线。

## 3. CubeMX / HAL 配置要求

目标 H743 工程需要先在 CubeMX 中完成外设配置，再放入驱动文件。

### 3.1 UART 配置

用于 RS422/RS485 的 UART 建议配置为：

```text
Mode: Asynchronous
BaudRate: 115200 初始值即可，RS485 启动时会通过 RS422_SetBaudRate() 改到目标波特率
WordLength: 8 Bits
Parity: None
StopBits: 1
Hardware Flow Control: None
TX/RX: Enable
FIFO: Disable
NVIC interrupt: Enable
```

本工程 RS485 默认波特率是 `2000000`，定义在 `rs485_speed_test.h`：

```c
#define RS485_SPEED_TEST_DEFAULT_BAUDRATE      2000000U
```

如果目标板的时钟配置导致 2 Mbps 误差较大，需要优先检查 UART kernel clock 和 `RS422_GetActualBaudRate()` 的返回值。

### 3.2 方向控制 GPIO

每路 422/485 收发器需要一个发送使能脚，也就是代码中的 `TX_EN`：

- GPIO 配置为 Output Push Pull。
- 上电默认写接收态，本工程是 `GPIO_PIN_RESET`。
- 如果目标板收发器发送使能为低有效，需要修改 `rs422_config[]` 中的 `tx_en_active`。

当前配置在 `rs422.c`：

```c
static const RS422_PortConfig rs422_config[RS422_PORT_COUNT] =
{
    { &huart1, TX_EN_1_GPIO_Port, TX_EN_1_Pin, GPIO_PIN_SET, USART1_IRQn },
    { &huart3, TX_EN_2_GPIO_Port, TX_EN_2_Pin, GPIO_PIN_SET, USART3_IRQn }
};
```

移植时主要改这里：

- `huart1` / `huart3` 改成目标工程实际使用的 UART 句柄。
- `TX_EN_x_GPIO_Port` / `TX_EN_x_Pin` 改成目标板方向脚。
- `GPIO_PIN_SET` 改成目标收发器的发送有效电平。
- `USARTx_IRQn` 改成对应 UART 的中断号。

### 3.3 中断处理

目标工程的 `stm32h7xx_it.c` 中，每个参与驱动的 UART 中断函数必须调用 HAL 中断入口：

```c
void USART1_IRQHandler(void)
{
    HAL_UART_IRQHandler(&huart1);
}

void USART3_IRQHandler(void)
{
    HAL_UART_IRQHandler(&huart3);
}
```

如果目标工程已经有 `HAL_UART_TxCpltCallback()`、`HAL_UART_RxCpltCallback()`、`HAL_UART_ErrorCallback()` 或 `HAL_UART_AbortReceiveCpltCallback()`，不能重复定义。需要把本工程 `rs422.c` 里的回调逻辑合并进去：

```c
void HAL_UART_TxCpltCallback(UART_HandleTypeDef *huart)
{
    RS422_UART_TxCpltCallback(huart);
    /* 目标工程其他 UART 模块回调 */
}

void HAL_UART_RxCpltCallback(UART_HandleTypeDef *huart)
{
    RS422_UART_RxCpltCallback(huart);
    /* 目标工程其他 UART 模块回调 */
}

void HAL_UART_ErrorCallback(UART_HandleTypeDef *huart)
{
    RS422_UART_ErrorCallback(huart);
    /* 目标工程其他 UART 模块回调 */
}
```

## 4. 工程移植步骤

### 4.1 拷贝文件

把第 1 节列出的核心文件复制到目标工程对应目录：

```text
Core/Inc/rs422.h
Core/Inc/rs422_protocol.h
Core/Inc/rs485_speed_test.h
Core/Inc/rs485_speed_test_wire.h
Core/Src/rs422.c
Core/Src/rs422_protocol.c
Core/Src/rs485_speed_test.c
```

然后在 Keil 工程中把 3 个 `.c` 文件加入编译分组，并确认 Include Path 包含 `Core/Inc`。

### 4.2 适配端口数量和 UART 映射

默认有两路 RS422 逻辑端口：

```c
typedef enum
{
    RS422_PORT_U9 = 0,
    RS422_PORT_U10,
    RS422_PORT_COUNT
} RS422_PortId;
```

如果目标工程只用一路，可以保留两个枚举但只使用 `RS422_PORT_U9`，这样改动最小。如果目标工程要扩展到三路或四路，需要同步修改：

- `RS422_PortId`
- `rs422_config[]`
- `RS422_PORT_COUNT` 关联的诊断数组和缓冲数组会自动跟随枚举数量。
- 上层业务中端口到从机地址、统计项的映射。

### 4.3 适配 RS485 主机物理总线

RS485 主机物理总线在 `rs485_speed_test.c` 顶部定义：

```c
#define RS485_SPEED_TEST_PORT               RS422_PORT_U9
```

如果目标板的 485 总线接在另一组 UART 上，改这个宏即可。例如接到第二路：

```c
#define RS485_SPEED_TEST_PORT               RS422_PORT_U10
```

如果目标板需要两条独立 485 物理总线，当前代码不能只改宏解决。当前设计是一条物理总线轮询多个从机地址，需要为每条物理总线拆独立状态机，或实例化 `rs485_speed_test` 的状态结构。

### 4.4 适配 RS485 从机地址

当前主机轮询两个从机：

```c
#define RS485_SPEED_TEST_MASTER_SLAVE_COUNT    2U

const uint8_t rs485_speed_test_master_slave_table[RS485_SPEED_TEST_MASTER_SLAVE_COUNT] =
{
    1U,
    2U
};
```

如果目标现场从机地址不同，改 `rs485_speed_test_master_slave_table[]`。如果从机数量变化，同时改 `RS485_SPEED_TEST_MASTER_SLAVE_COUNT`，并检查 `RS485_SpeedTestInstanceId` 和 `RS485_SpeedTestInstanceToSlaveIndex()` 是否仍然满足上层界面映射。

### 4.5 选择 RS422 或 RS485 模式

如果移植 `host_comm`，模式通过编译宏选择，不是运行时动态切换：

```c
#define HOST_COMM_TEST_MODE_RS422              0U
#define HOST_COMM_TEST_MODE_RS485              1U
#define HOST_COMM_TEST_MODE_DEFAULT            HOST_COMM_TEST_MODE_RS422
```

需要 RS485 主机测试时改为：

```c
#define HOST_COMM_TEST_MODE_DEFAULT            HOST_COMM_TEST_MODE_RS485
```

修改后必须重新编译并下载。

## 5. 初始化和主循环调用

### 5.1 只使用 RS422 驱动和协议层

初始化顺序：

```c
MX_GPIO_Init();
MX_USART1_UART_Init();
MX_USART3_UART_Init();

RS422_Init();
```

发送一帧 128 字节业务数据：

```c
uint8_t tx_data[RS422_PROTOCOL_DATA_LENGTH] = {0U};

(void)RS422_ProtocolSend(RS422_PORT_U9,
                         1U,
                         (uint8_t)RS422_PROTOCOL_TYPE_55,
                         tx_data,
                         200U);
```

主循环中轮询接收：

```c
RS422_ProtocolPacket packet;
RS422_ProtocolStatus status;

status = RS422_ProtocolPoll(RS422_PORT_U9, &packet);
if (status == RS422_PROTOCOL_STATUS_OK)
{
    /* 使用 packet.id、packet.data_type、packet.data */
}
```

### 5.2 使用 RS485 主机测试

初始化顺序：

```c
MX_GPIO_Init();
MX_USART1_UART_Init();
MX_USART3_UART_Init();

RS422_Init();
RS485_SpeedTest_Init();
```

开始测试：

```c
RS485_SpeedTest_Start(RS485_SPEED_TEST_INSTANCE_UART1, 2000000U); /* 从机地址 1 */
RS485_SpeedTest_Start(RS485_SPEED_TEST_INSTANCE_UART3, 2000000U); /* 从机地址 2 */
```

主循环必须持续调用：

```c
while (1)
{
    RS485_SpeedTest_Run();
}
```

停止测试：

```c
RS485_SpeedTest_Stop(RS485_SPEED_TEST_INSTANCE_UART1);
RS485_SpeedTest_Stop(RS485_SPEED_TEST_INSTANCE_UART3);
/* 或 */
RS485_SpeedTest_StopAll();
```

读取统计：

```c
RS485_SpeedTestStats stats;

RS485_SpeedTest_GetStats(RS485_SPEED_TEST_INSTANCE_UART1, &stats);
```

## 6. 协议格式

RS422 协议层固定帧长 `135` 字节：

| 字段      | 长度     | 说明                                         |
| --------- | -------- | -------------------------------------------- |
| Header    | 2 字节   | 固定`0x0F 0xF0`                            |
| ID        | 1 字节   | 节点或板卡接口号                             |
| Data Type | 1 字节   | 只允许`0x00`、`0xFF`、`0x55`、`0xAA` |
| Direction | 1 字节   | `0` 表示发送，`1` 表示接收               |
| Length    | 1 字节   | 固定`128`                                  |
| Data      | 128 字节 | 业务数据                                     |
| Checksum  | 1 字节   | 从 ID 到 Data 末尾的 8 位累加和              |

RS485 测速载荷放在 `Data[128]` 内，前几个关键字段为：

| 偏移    | 字段        | 说明                                |
| ------- | ----------- | ----------------------------------- |
| 0       | Magic0      | 固定`0x52`                        |
| 1       | Magic1      | 固定`0x35`                        |
| 2       | Version     | 当前为`2`                         |
| 3       | Command     | `DATA`、`ECHO`、`SET_BAUD` 等 |
| 4       | Index       | 波特率表索引                        |
| 8       | Baud        | 小端 32 位波特率                    |
| 12      | Sequence    | 小端 32 位序号                      |
| 20      | Actual Baud | 小端 32 位实际波特率                |
| 24..127 | Pattern     | 按序号生成的校验 pattern            |

## 7. 调试和验收

建议按以下顺序验证。

### 7.1 编译检查

常见编译错误和处理：

| 现象                                | 处理                                                                     |
| ----------------------------------- | ------------------------------------------------------------------------ |
| 找不到`huart1` / `huart3`       | 确认`usart.h` 已包含，目标工程 UART 句柄名称和 `rs422_config[]` 一致 |
| 找不到`TX_EN_1_Pin`               | 在 CubeMX 里给方向脚设置 User Label，或直接改成目标工程已有 GPIO 宏      |
| 重复定义`HAL_UART_RxCpltCallback` | 合并回调，不要在多个`.c` 文件中重复定义 HAL weak callback              |
| 找不到`HostComm_UART_*`           | 未移植`host_comm` 时，删除 `rs422.c` 中对应转发调用                  |

### 7.2 RS422 单路收发验证

用一组 422 收发器闭环，或两块板互连，重点观察：

```c
rs422_diag_start_receive_status[port]  /* 0 表示 HAL_OK */
rs422_diag_tx_complete_count[port]
rs422_diag_rx_complete_count[port]
rs422_diag_error_count[port]
rs422_protocol_rx_good_frames[port]
rs422_protocol_rx_error_rate_ppm[port]
```

判断方式：

- `tx_complete_count` 增加，说明 MCU 已经完成 UART 发送。
- `rx_complete_count` 增加，说明 UART RX 已收到字节。
- `rx_good_frames` 增加，说明协议帧完整且校验正确。
- RX 有字节但 good frame 不增加，优先检查 A/B 极性、波特率、帧格式和 GND。

### 7.3 RS485 主机验证

RS485 现场验证注意：

- 总线两端建议加 120 欧姆终端电阻。
- 多节点必须共地。
- A/B 极性不确定时，用低波特率或示波器先确认。
- 当前主机一次只等待一个从机回包，超时时间为 `RS485_SPEED_TEST_FRAME_TIMEOUT_MS`，当前 `120ms`。

关键观察变量：

```c
rs485_speed_test_current_slave_id
rs485_speed_test_current_slave_index
rs485_speed_test_tx_frames
rs485_speed_test_rx_frames
rs485_speed_test_echo_frames
rs485_speed_test_timeout_count
rs485_speed_test_ignored_node_frames
rs485_speed_test_last_status
rs485_speed_test_last_hal_status
```

判断方式：

- `tx_frames` 增加但 `rx_frames` 不增加：从机没有回包，查接线、从机地址、方向脚和波特率。
- `rx_frames` 增加但 `echo_frames` 不增加：回包 ID、命令、序号或 payload pattern 不匹配。
- `ignored_node_frames` 增加：收到的回包 ID 不是当前正在等待的从机地址。
- `last_hal_status` 非 `HAL_OK`：查 UART 初始化、发送队列是否满、IRQ 是否被禁用。

## 8. 移植时最容易踩的坑

1. 不要把 U10 当成 `USART2`。本工程 U10 已确认是 `USART3`。
2. 不要直接复制 `usart.c` 和 `gpio.c` 覆盖目标工程。目标板引脚通常不同，应由 CubeMX 生成后只改驱动配置表。
3. `TX_EN` 有效电平必须按目标收发器确认。电平反了会表现为能发送中断完成，但总线上没有有效数据，或一直收不到回包。
4. HAL UART 回调只能有一份定义。目标工程已有串口业务时，必须合并回调分发。
5. 当前 RS485 主机是“一条物理总线 + 多个从机地址”的设计，不是“两条 UART 同时跑两条 485 总线”。
6. `HostComm` 只是上位机/串口屏控制层，不是 RS422/RS485 驱动的必需部分。
7. 当前 RS485 代码实现的是 H743 主机侧测试逻辑。从机侧需要能识别 `DATA` 并回 `ECHO`，否则主机会一直超时。
8. H743 开 D-Cache 不影响当前中断单字节收发，因为没有用 DMA。如果后续改成 DMA，需要重新处理缓存一致性。

## 9. 建议交付给移植方的确认清单

- 已确认目标板 RS422/RS485 使用的 UART、TX/RX 引脚和 AF 复用。
- 已确认方向控制脚、默认接收态和发送有效电平。
- 已确认 UART IRQ 已开启，`USARTx_IRQHandler()` 调用 `HAL_UART_IRQHandler()`。
- 已合并 HAL UART 回调，没有重复定义。
- 已把 `rs422_config[]` 改成目标硬件资源。
- 已按现场从机地址修改 `rs485_speed_test_master_slave_table[]`。
- 已确认 RS485 是单总线轮询还是多物理总线，如果是多物理总线，需要额外拆分状态机。
- 已完成 RS422 单路协议帧收发验证。
- 已完成 RS485 从机地址 1、地址 2 的轮询和统计验证。
