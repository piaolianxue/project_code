整体思路
工程入口在 [main.c (line 79)]。
启动流程是：
开 I-Cache / D-Cache。
HAL_Init() 初始化 HAL 和系统节拍。
SystemClock_Config() 配系统时钟和外设时钟。
初始化 GPIO、SPI、UART：
MX_GPIO_Init()、MX_SPI3_Init()、MX_USART1/2/3_UART_Init()、MX_SPI2_Init()、MX_SPI4_Init()。
初始化业务驱动：
ADS7951_Init()、RS422_Init()、SSI422_Init()。
上电后先读一次 ADC 和 SSI422。
主循环里目前持续读取 U16/U18 两路 SSI422 编码器值。
所以现在主程序还是偏“底层验证/驱动搭建阶段”，业务逻辑没有复杂状态机，重点是先把外设链路跑通。
模块分层
现在可以理解成三层：
第一层是 CubeMX/HAL 生成层：
[gpio.c (line 41)]：配置片选、方向控制、DE/RE、TX_EN 等 GPIO。
[spi.c (line 35)]：配置 SPI2、SPI3、SPI4。
[usart.c (line 35)]：配置 USART1、USART2、USART3。
[main.h (line 61)]：集中定义 GPIO 引脚宏。
第二层是设备驱动层：
[ads7951.c (line 12)]：两片 ADS7951 ADC。
[ssi422.c (line 32)]：两路 SSI422 编码器读取。
[rs422.c (line 225)]：两路 RS422 串口通信。
第三层是应用层：
[main.c (line 116)]：调用各驱动读数，目前主要保存到全局变量，方便 DAPLink/Keil 调试看。
ADS7951 模块
ADS7951 是 ADC 采样模块，走 SPI3。
核心思路：
SPI3 是公共总线。
U6 和 U8 两片 ADS7951 用不同片选脚区分。
驱动里把“构造命令、拉低片选、SPI 交换、拉高片选、取 12 位结果”封装起来。
对外提供：ADS7951_U6_ReadChannel()
ADS7951_U8_ReadChannel()
ADS7951_U6_ReadAll()
ADS7951_U8_ReadAll()

这里用了一个比较好的封装方式：底层通用函数 ADS7951_ReadRaw() 接收片选端口和通道号，上层再封装成 U6/U8 专用接口。这样以后加第三片 ADC，也不用重写读数逻辑。
SSI422 模块
SSI422 是编码器读数模块，走 SPI2 和 SPI4。
当前对应关系是：
U16 -> SPI4
U18 -> SPI2
核心思路：
每个 SSI422 口都有一组 SPI 句柄、DE、RE 控制脚。
用 SSI422_PortConfig 表描述硬件资源。
读数前先切到编码器读取模式。
SPI 发送 dummy 数据，同时接收编码器返回值。
SSI422_ReadBits() 会把多个 16 位 word 拼成一个 32 位原始值，再按实际 bit 数右对齐。
主循环现在一直做：
ssi_u16_status = SSI422_U16_ReadBits(16U, &ssi_u16_value);
ssi_u18_status = SSI422_U18_ReadBits(16U, &ssi_u18_value);
所以当前程序运行时，U16/U18 是持续刷新的。
RS422 模块
RS422 是目前做得最完整的通信驱动。
当前硬件对应关系：
U9 -> USART1
U10 -> USART3
这里之前排查后已经修正过：U10 实际接的是 UART3，不是 UART2。
RS422 驱动的核心思路是：
每个端口有独立配置：UART 句柄、TX_EN 控制脚、发送有效电平。
每个端口有独立状态：接收环形缓冲、发送环形缓冲、忙标志。
接收使用 HAL_UART_Receive_IT() 单字节中断接收。
收到一个字节后进环形缓冲，再重新挂下一字节接收。
发送使用中断方式逐字节发送，发送完成后自动继续取下一个字节。
发完后自动切回接收模式。
对外主要接口是：
RS422_Transmit_IT()：非阻塞发送。
RS422_Transmit()：阻塞等待发送完成。
RS422_Available()：看接收缓存里有多少字节。
RS422_Read()：从接收缓存取数据。
RS422_ClearRx()：清接收缓存。
这个模块已经不是简单 HAL 包装了，它有自己的收发队列和诊断变量，后面接协议会比较方便。

### RS422 通讯协议开发

本次在原有 `rs422.c` 底层收发驱动上新增了一层 RS422 协议封装，代码文件为：

- `Core/Inc/rs422_protocol.h`
- `Core/Src/rs422_protocol.c`

协议帧格式确定为：

| 字段     | 长度     | 说明                                         |
| -------- | -------- | -------------------------------------------- |
| 包头     | 2 字节   | 固定为`0x0F 0xF0`，对应需求中的 `0x0FF0` |
| ID       | 1 字节   | 板子不同对外接口号                           |
| 数据类型 | 1 字节   | 只允许`0x00`、`0xFF`、`0x55`、`0xAA` |
| 收发指示 | 1 字节   | `0` 表示发送，`1` 表示接收               |
| 数据长度 | 1 字节   | 固定为`128`                                |
| 数据区   | 128 字节 | 实际业务数据                                 |
| 校验位   | 1 字节   | 从 ID 到数据区做 8 位累加校验                |

完整协议帧长度为 `135` 字节。这样做的好处是包头清晰、长度固定、解析简单，后面在串口字节流中只需要先同步包头，再按固定长度收满一帧即可。

### 协议层主要接口

`rs422_protocol.c` 对外提供的核心函数如下：

- `RS422_ProtocolBuildFrame()`：按照协议格式组装 135 字节完整数据帧。
- `RS422_ProtocolSend()`：组帧后调用底层 `RS422_Transmit()` 发送。
- `RS422_ProtocolDecodeFrame()`：检查包头、数据类型、方向、长度和校验位，并解析为结构体。
- `RS422_ProtocolPoll()`：从底层 RS422 接收缓存中轮询字节流，自动寻找包头并解析完整协议包。
- `RS422_ProtocolClearStats()`：清空指定端口的协议统计。
- `RS422_ProtocolGetStats()`：读取指定端口的接收统计。

接收统计中保留了以下信息，方便在 Keil Watch 中直接观察：

- 接收总字节数
- 正确帧数
- 错误帧数
- 错误字节数
- 包头同步丢弃字节数
- 误码率，单位为 ppm

### RS422 测试与示例代码

`RS422_UART1_EXAMPLE_ENABLE` 用于单独演示串口1的协议收发调用。串口1在当前工程中对应 `RS422_PORT_U9`，也就是 `USART1 / U9 / J12`。打开后会周期发送一帧协议数据，同时轮询接收端口中的协议包。

### 串口发送和接收调用

当前业务层推荐优先调用协议层接口，因为协议层已经处理了 `0x0FF0` 包头、135 字节固定帧长、数据类型检查和累加校验。

发送一帧 128 字节业务数据：

```c
uint8_t tx_data[RS422_PROTOCOL_DATA_LENGTH] = {0U};

/* 填充 tx_data[0..127] 后发送，U9 对应 USART1/J12。 */
HAL_StatusTypeDef tx_status =
    RS422_ProtocolSend(RS422_PORT_U9,
                       1U,
                       (uint8_t)RS422_PROTOCOL_TYPE_55,
                       tx_data,
                       200U);
```

接收时放在主循环里轮询：

```c
RS422_ProtocolPacket rx_packet = {0U};
RS422_ProtocolStatus rx_status;

rx_status = RS422_ProtocolPoll(RS422_PORT_U9, &rx_packet);
if (rx_status == RS422_PROTOCOL_STATUS_OK)
{
    /* 收到完整协议帧后，在这里使用 rx_packet.id、rx_packet.data_type、rx_packet.data。 */
}
```

如果只是临时调试裸串口字节，不走协议层，也可以直接调用底层 RS422 驱动：

```c
uint8_t raw_tx[] = {0x11U, 0x22U, 0x33U};
uint8_t raw_rx[32] = {0U};
uint16_t raw_rx_len;

/* 阻塞等待发送完成。 */
(void)RS422_Transmit(RS422_PORT_U9, raw_tx, sizeof(raw_tx), 100U);

/* 非阻塞发送，只把数据放入发送队列。 */
(void)RS422_Transmit_IT(RS422_PORT_U9, raw_tx, sizeof(raw_tx));

/* 查询接收缓存，有数据就读出来。 */
if (RS422_Available(RS422_PORT_U9) > 0U)
{
    raw_rx_len = RS422_Read(RS422_PORT_U9, raw_rx, sizeof(raw_rx));
}
```

端口选择保持当前映射：`RS422_PORT_U9` 是 `USART1 / U9 / J12`，`RS422_PORT_U10` 是 `USART3 / U10 / J13`。

| 帧头     | 1 字节 | 固定`0xEE`                                                     |

### 上位机使用顺序

### 测试规则

- 串口1使能开着，就测串口1。
- 串口3使能开着，就测串口3。

- `TX` = 实际送入 U9/U10 协议帧的总字节数
- `RX` = 实际收到的协议层总字节数

### 使能与启动问题修正

本次排查发现两个现象：

- 串口未使能时，上位机仍能看到统计数据变化。
- 点击开始后，485 侧没有收到回包。

代码原因：

- 软件默认把串口1到串口4使能都置为 `1`，导致上位机没有明确使能时，开始测试仍可能启动串口1/串口3。
- 测试运行后再关闭使能，只改了使能标志，没有停止已经运行的 485 测试实例。
- 上传统计时没有判断使能状态，未使能端口也会上传旧的 TX/RX/误码率计数。
- 开始/停止按钮原来只接受动作值 `0x00`，如果现场屏幕发送 `0x01`，H743 会拒绝命令，测试不会启动。

修正方式：

- 上电和 `HostComm_Init()` 后，串口1到串口4默认全部未使能。
- 只有上位机打开串口1或串口3使能后，开始测试才会启动对应 485 实例。
- 禁用串口1或串口3时，立即停止对应 485 测试实例。
- 这一版先把未使能端口处理成上传 0，后续已修正为不发送未使能端口统计帧。
- 开始/停止按钮兼容 `0x00` 和 `0x01` 两种动作值，适配不同串口屏按钮事件配置。

### 使能上传策略修正

上一版把未使能端口处理成“上传 0”，但这对上位机来说仍然是在接收数据，也会刷新显示控件。

本次改为：

- 未使能端口不发送 TX/RX/误码率三帧。
- 只有串口1或串口3使能打开后，才上传对应端口统计。
- 现场串口屏的使能开关动作按 `0x00` 作为打开、`0x01` 作为关闭处理；如后续屏幕工程改回相反极性，只需要调整 `HOST_COMM_ENABLE_ACTION_ON` 和 `HOST_COMM_ENABLE_ACTION_OFF` 两个宏。

- 未保留使能按钮极性转换、未使能端口不上传等后续试错策略。

## 2026年7月16日

### RS485 主机双从机轮询固定版本

本次固定的是 H743 主机在 RS485 模式下挂载两个从机的通讯方式。主机当前使用一条物理 485 总线发送测试帧，每次只对一个从机地址发包，不同时向两个从机发包。

### 硬件和逻辑映射

- `J12 / U9 / USART1`：当前 RS485 主机物理发送和接收总线，代码中对应 `RS422_PORT_U9`。
- `J13 / U10 / USART3`：硬件上仍然对应串口3，但当前 RS485 主机双从机轮询不再把它作为第二条独立物理 485 总线发送。
- 上位机“串口1”使能：对应 RS485 从机地址 `1`。
- 上位机“串口3”使能：对应 RS485 从机地址 `2`。

### 轮询规则

- 只使能串口1时，主机只向从机地址 `1` 发包。
- 只使能串口3时，主机只向从机地址 `2` 发包。
- 串口1和串口3都使能时，主机按 `地址1 -> 地址2 -> 地址1 -> 地址2` 的顺序轮询。
- 每次发包后，主机会等待对应从机回包；收到正确回包后立即切到下一个使能从机。
- 如果当前从机没有回包，等待 `RS485_SPEED_TEST_FRAME_TIMEOUT_MS`，当前值为 `120ms`，超时后再切到下一个使能从机。

### 地址和数据帧

主机发送时使用 `RS422_ProtocolSend_IT()`，目标节点 ID 使用当前轮询到的从机地址：

```c
RS422_ProtocolSend_IT(RS485_SPEED_TEST_PORT,
                       target_node_id,
                       RS485_SPEED_TEST_DATA_TYPE,
                       payload);
```

当前从机地址表为：

```c
const uint8_t rs485_speed_test_master_slave_table[RS485_SPEED_TEST_MASTER_SLAVE_COUNT] =
{
    1U,
    2U
};
```

主机接收回包时会检查回包 ID，只接受当前正在等待的从机地址。不是当前从机地址的回包会被计入 `rs485_speed_test_ignored_node_frames`，不会推进轮询状态。

### 统计口径

今天同步修正了 UART1 / UART3 的统计口径，避免两个逻辑串口共用同一条物理总线统计：

- 未使能的逻辑串口统计保持 `0`。
- 串口1统计只来自从机地址 `1` 的 `tx_frames` / `rx_frames`。
- 串口3统计只来自从机地址 `2` 的 `tx_frames` / `rx_frames`。
- `tx_bytes = 对应从机 tx_frames * RS422_PROTOCOL_FRAME_LENGTH`。
- `rx_bytes = 对应从机 rx_frames * RS422_PROTOCOL_FRAME_LENGTH`。

这样现场只打开串口1时，只有串口1显示发送/接收统计；只打开串口3时，只有串口3显示发送/接收统计。

### RS422 / RS485 切换方式

当前工程的 RS422 和 RS485 不是上位机运行时动态切换，而是编译时通过宏选择。切换位置在 `Core/Inc/host_comm.h`：

```c
#define HOST_COMM_TEST_MODE_RS422              0U
#define HOST_COMM_TEST_MODE_RS485              1U
#define HOST_COMM_TEST_MODE_DEFAULT            HOST_COMM_TEST_MODE_RS485
```

当前固定版本默认编译为 RS485：

```c
#define HOST_COMM_TEST_MODE_DEFAULT            HOST_COMM_TEST_MODE_RS485
```

如果现场要切回 RS422 测试，把默认模式改成：

```c
#define HOST_COMM_TEST_MODE_DEFAULT            HOST_COMM_TEST_MODE_RS422
```

修改后需要重新 Keil 编译并下载到板子，新的模式才会生效。

两种模式的行为区别：

- RS485 模式：串口1和串口3是两个逻辑从机地址，实际走同一条 485 物理总线轮询。
- RS422 模式：串口1走 `J12 / U9 / USART1`，串口3走 `J13 / U10 / USART3`，两路物理串口分别发送测试帧。

### RS422 未使能上传修正

现场切到 RS422 收发模式后发现：串口1未使能时，上位机仍然会持续收到串口1 TX/RX/误码率上传帧。排查后确认 RS422 测试发送本身已经受 `host_comm_uart1_enabled` 控制，未使能时不会主动发送测试帧；问题在 `HostComm_SendStats()` 仍然每秒固定上传 1/2/3/4 四个串口的统计帧。

本次修正为：

- 增加 `HostComm_IsPortEnabled()`，统一读取串口1到串口4使能状态。
- 增加 `HostComm_SendPortStatsIfEnabled()`，只有端口使能时才发送 TX/RX/误码率三帧。
- `HostComm_SendStats()` 改为全部走使能门控，未使能端口不再向上位机发送统计帧。
- 该规则同时适用于 RS422 和 RS485 模式。

修正后，RS422 模式下串口1不使能时，上位机不会再持续收到串口1统计数据；只有打开串口1使能后，才会上报串口1统计。

### RS422 串口3无接收排查记录

现场切到 RS422 收发模式后，串口1有数据，串口3收不到数据。本次排查先通过 Keil Watch 判断软件链路状态：

```text
host_comm_uart3_enabled              = 0x01
host_comm_test_running               = 0x01
rs422_diag_start_receive_status[1]   = 0x00000000
rs422_diag_tx_total_bytes[1]         = 0x0000032A
rs422_diag_tx_complete_count[1]      = 0x0000032A
rs422_diag_rx_complete_count[1]      = 0x00000000
rs422_diag_error_count[1]            = 0x00000000
rs422_protocol_rx_total_bytes[1]     = 0x00000000
rs422_protocol_rx_good_frames[1]     = 0x00000000
```

其中索引 `[1]` 对应 `RS422_PORT_U10`，也就是串口3 / J13 / USART3。根据这些变量可以确认：

- 串口3已经被上位机使能。
- 测试已经开始运行。
- USART3 的中断接收启动成功，`start_receive_status[1] = HAL_OK`。
- 串口3发送计数和发送完成计数都在增加，说明 MCU 侧认为 U10/USART3 已经完成发送。
- 串口3接收完成计数一直为 0，说明 USART3_RX 没有收到任何字节。

随后用示波器测量 J13 的 3/4 脚，发现没有波形。结合 Watch 结果，问题范围收敛为 MCU 串口3发送链路到 J13 外部连接之间的物理链路，而不是上位机协议或 `HostComm` 使能逻辑。

最终现场确认原因是连接线序有问题。调整线序后，应重点复核以下变量：

```c
rs422_diag_rx_complete_count[1]
rs422_protocol_rx_total_bytes[1]
rs422_protocol_rx_good_frames[1]
```

如果 `rs422_diag_rx_complete_count[1]` 开始增加，说明 USART3_RX 已经收到字节；如果 `rs422_protocol_rx_good_frames[1]` 也增加，说明串口3完整协议帧接收正常。若 RX 计数增加但 good frame 不增加，再继续检查 A/B 极性、波特率和帧格式。

## 2026年7月20日

### I2C / CANFD 主机通讯接口

本次新增的是 H743 作为主机与从机通讯的 I2C 和 CANFD 接口。这个接口层只负责把外设、协议帧和基础 API 打通，不改变现有主机与上位机屏幕的通讯协议，也不改变 RS422 / RS485 测试模式的选择方式。

当前实现中，I2C / CANFD 与 `host_comm` 是相互独立的。主循环会轮询 `SlaveComm_Poll()` 接收 CANFD 响应，但不会自动周期发送 `PING`、不会自动轮询寄存器，也不会主动向屏幕新增上传帧。后续如果要从屏幕按钮或上位机命令触发 I2C/CANFD 操作，再在 `host_comm` 中调用 `slave_comm` 接口即可。

### 硬件映射

- I2C 使用 `PB8 / PB9`，代码中配置为 `I2C1`。
- `PB8 / PB9` 配置为 `GPIO_MODE_AF_OD`、`GPIO_PULLUP`、`GPIO_AF4_I2C1`。
- CANFD 使用 `PD0 / PD1`，代码中配置为 `FDCAN1`。
- `PD0 / PD1` 配置为 `GPIO_MODE_AF_PP`、`GPIO_NOPULL`、`GPIO_SPEED_FREQ_VERY_HIGH`、`GPIO_AF9_FDCAN1`。

新增外设文件：

- `Core/Inc/i2c.h`
- `Core/Src/i2c.c`
- `Core/Inc/fdcan.h`
- `Core/Src/fdcan.c`

新增应用层通讯封装：

- `Core/Inc/slave_comm.h`
- `Core/Src/slave_comm.c`

### I2C 协议和接口

从机 7 位地址为：

```c
#define SLAVE_COMM_I2C_ADDRESS_7BIT        0x43U
#define SLAVE_COMM_I2C_ADDRESS             (SLAVE_COMM_I2C_ADDRESS_7BIT << 1U)
```

这里 `SLAVE_COMM_I2C_ADDRESS_7BIT` 是协议地址，`SLAVE_COMM_I2C_ADDRESS` 是传给 STM32 HAL I2C 主机接口的左移后地址。

从机侧协议规则：

- I2C2 和 I2C4 共用同一套寄存器表。
- I2C 写：第 1 字节是寄存器地址，后续字节写入连续寄存器。
- I2C 读：从当前寄存器地址开始连续读。

主机侧封装接口：

```c
HAL_StatusTypeDef SlaveComm_I2C_WriteRegs(uint8_t reg,
                                          const uint8_t *data,
                                          uint16_t length);

HAL_StatusTypeDef SlaveComm_I2C_ReadRegs(uint8_t reg,
                                         uint8_t *data,
                                         uint16_t length);
```

`SlaveComm_I2C_WriteRegs()` 内部会把寄存器地址和数据拼成一个发送缓冲区，第 1 字节为 `reg`，后续为数据。

`SlaveComm_I2C_ReadRegs()` 内部会先发送 1 字节起始寄存器地址，再调用 `HAL_I2C_Master_Receive()` 读取指定长度数据。

当前限制：

- 单次 I2C 读写最大长度为 `64` 字节。
- I2C 超时时间为 `100ms`。
- 长度非 0 但数据指针为空时返回 `HAL_ERROR`。

调用示例：

```c
uint8_t write_data[2] = {0x12U, 0x34U};
uint8_t read_data[4] = {0U};

/* 从寄存器 0x10 开始连续写入 2 字节。 */
(void)SlaveComm_I2C_WriteRegs(0x10U, write_data, sizeof(write_data));

/* 从寄存器 0x20 开始连续读取 4 字节。 */
(void)SlaveComm_I2C_ReadRegs(0x20U, read_data, sizeof(read_data));
```

### CANFD 协议和接口

CANFD 使用扩展 ID：

```c
#define SLAVE_COMM_CAN_COMMAND_ID          0x18EF4301UL
#define SLAVE_COMM_CAN_RESPONSE_ID         0x18EF0143UL
```

数据格式：

| 字节位置 | 说明 |
| -------- | ---- |
| Byte0 | `sequence`，主机发送时自动递增 |
| Byte1 | `command` |
| Byte2.. | 命令参数 |

支持命令：

| 命令 | 数值 | 主机侧接口 |
| ---- | ---- | ---------- |
| `PING` | `0x01` | `SlaveComm_CanPing()` |
| `READ_REG` | `0x02` | `SlaveComm_CanReadReg()` |
| `WRITE_REG` | `0x03` | `SlaveComm_CanWriteReg()` |
| `SET_CAN_MODE` | `0x04` | `SlaveComm_CanSetCanMode()` |
| `CLEAR_FLAGS` | `0x05` | `SlaveComm_CanClearFlags()` |

主机侧封装接口：

```c
HAL_StatusTypeDef SlaveComm_CanPing(void);
HAL_StatusTypeDef SlaveComm_CanReadReg(uint8_t reg, uint8_t length);
HAL_StatusTypeDef SlaveComm_CanWriteReg(uint8_t reg,
                                        const uint8_t *data,
                                        uint8_t length);
HAL_StatusTypeDef SlaveComm_CanSetCanMode(uint8_t mode);
HAL_StatusTypeDef SlaveComm_CanClearFlags(uint16_t flags);
```

CANFD 发送帧由 `SlaveComm_CanSendCommand()` 内部统一封装。它会自动填写 Byte0 的 sequence，Byte1 填命令码，Byte2 之后复制参数。

`SlaveComm_Poll()` 会从 FDCAN RX FIFO0 读取响应，只接受扩展 ID `0x18EF0143` 的数据帧。收到的最后一帧会保存到以下 Watch 变量中：

- `slave_comm_last_can_id`
- `slave_comm_last_can_sequence`
- `slave_comm_last_can_command`
- `slave_comm_last_can_length`
- `slave_comm_last_can_data`

当前 FDCAN 配置集中在 `Core/Src/fdcan.c`：

- 帧格式：`FDCAN_FRAME_FD_BRS`
- 模式：`FDCAN_MODE_NORMAL`
- RX FIFO0：4 个元素，每个 64 字节
- TX FIFO Queue：4 个元素，每个 64 字节
- 过滤器：只接收扩展 ID `0x18EF0143`
- 全局过滤：拒收其它标准帧、扩展帧和远程帧

当前位时序参数：

| 段 | Prescaler | SJW | TimeSeg1 | TimeSeg2 |
| -- | --------- | --- | -------- | -------- |
| 仲裁段 | `24` | `1` | `16` | `3` |
| 数据段 | `6` | `1` | `12` | `3` |

在当前 `SystemClock_Config()` 中，PLL 参数为 `PLLM=5`、`PLLN=192`、`PLLQ=2`，`fdcan.c` 选择 `RCC_FDCANCLKSOURCE_PLL`。如果板上 HSE 仍按 25MHz 使用，则 FDCAN 内核时钟为 480MHz，对应仲裁段约 1Mbps、数据段约 5Mbps。后续如果从机要求其它 CANFD 波特率，优先修改 `Core/Src/fdcan.c` 中这组位时序参数。
